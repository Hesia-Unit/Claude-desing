## Jetson Runtime Residuals To Revisit

These are not invented; they come from observed behavior and logs.

1. The file replay loop support is implemented locally and deployed.

2. A fast-fail transport shutdown patch was also deployed so that confirmed
   transport failure now stops the send / telemetry / video loops faster.

3. The server simultaneously continued to receive `VIDEO_DATA ok` and telemetry
   updates during earlier failure storms, which suggests:

- either the drone error path is over-reporting transient write failures
- or multiple transport attempts overlap before disconnection is fully handled
- or the logging chronology between systemd/journal and per-session logs is
  partially misleading

4. `VoltageGlitchProtection` currently initializes and writes its log on the
   Jetson. The older "cannot open log file" problem is no longer the dominant
   issue.

5. `Clock jamming` false positives were previously a concern and should remain
   on the radar, but they were not the blocking issue in the most recent Jetson
   pass.

Recommendation for future agent:

- inspect the drone send path, especially `send_message()`,
  `enqueue_message()`, `send_loop()`, and the frame callback path in
  `drone_network.cpp`
- correlate drone-side failures with server-side session continuity
- reduce log storms if the failures are transient and expected under backpressure
